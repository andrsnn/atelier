# Atelier reference

The details behind the [README](../README.md): configuration, the terminal UI,
ways to run it, and how the pieces work. Deployment on an always-on host is in
[deploy.md](deploy.md).

## Configuring

### Machines (loops) - `machines/*.yaml`

A machine is an ordered list of states, each:

```
{ id, name, prompt, tools[], gate?, rejectTo?, optional?, offPath?/returnTo?,
  maxTurns?, model?, modes?, caseInput?, output?, watchable? }
```

plus `settings` (loop cap, turn budget, timeout, run `modes`,
`selfLearn`/`principles`/`reflectPrompt`/`evolveStructure`). Edit a file and
restart, or edit in the UI (`/machines`) or the terminal editor - reorder,
attach tools, set gates and loop-backs, pick a per-phase model, or let AI
draft/chat-edit the loop. The `version:` field reseeds the DB when bumped.

Four ship by default:

- **repo-pipeline** - the SDLC loop: `PRD → Spec → Mock → Build → Review →
  Rigorous QA → Acceptance → Open PR`, with an off-path `Deep-dive` leaf and
  three run modes (Rigorous / Smoke / Mock API).
- **goal-loop** - `Brief → Make → Evaluate ↺ → Deliver`, general-purpose.
- **goal-seeker** - `Anchor → Attempt → Evaluate ↺ Unblock`, self-unblocking.
- **image-collector** - `Brief → Collect → Curate ↺ → Package`.

### Tools - `app/lib/engine/tools.ts`

What a state can be granted: `read_file`, `list_directory`, `search_code`,
`run_command`, `write_file`, `edit_file`, `display_artifact` (surface a
reviewable deliverable - markdown/html/image/video/file), `start_dev_server`,
`authenticate` (log into auth-walled apps once), `screenshot_page`,
`record_walkthrough` (scripted MP4), `analyze_image` (the vision helper),
`gemini_eval` (image *and* video judge), `test_drive`, `watch_pr` (CI +
rebase), `record_principles`. `request_approval` / `reject` / `complete` are
always available - they end a phase.

### Projects - `atelier.projects.json`

Gitignored; copy the `.example`. Each entry is a repo path + base branch, with
optional `devCommand`, `setupCommand`, and an `auth` recipe (login path +
test-credential env vars) so QA can sign in.

### Models

Pick the primary + vision helper per loop in the New-loop form; any state can
pin its own `model:` in the machine YAML, and an overriding phase runs isolated
without disturbing the main agent thread. Every model - Claude-native or
Ollama-Cloud - drives the full Claude Code harness: Claude models run the
`claude` CLI; Ollama models run via `ollama launch claude`.

### Env - `.env.local` (see `.env.example`)

`OLLAMA_API_KEY` (required) plus optional `FACTORY_WORKSPACES_DIR`,
`ACCESS_PASSWORD` (gate + share the board), `FACTORY_INTERNAL_SECRET`
(generated by `setup.sh`), `CHROME_PATH`, `FACTORY_PUBLIC_HOST`,
`FACTORY_MAX_CONCURRENT_RUNS` (default 1), `CLAUDE_BIN`, and `GEMINI_API_KEY`
(the `gemini_eval` video judge). The terminal UI reads `ATELIER_URL` /
`ATELIER_USER` / `ATELIER_PASSWORD`.

### Claude Code workflows - `.claude/workflows/*.js`

Any [Claude Code workflow](https://code.claude.com/docs/en/workflows) in the
repo is loaded and visualized alongside the YAML machines (or paste one via
**Import Claude workflow** on `/machines`). The board parses it statically -
never executing it - and shows its phases, agents, and
`parallel`/`pipeline`/`loop` shape; a phase's pinned `model` shows too.

## The terminal interface

`atelier` is the same factory from a terminal - a full-screen TUI with zero
dependencies, a pure client of the same HTTP API as the board:

```bash
npm run tui                                                # this machine's factory
node cli/atelier.mjs --url http://host:7777 --user maria   # someone else's
```

- **Board** - every loop in flight with a mini state-rail, in a feed or
  status-column layout; sort, archive, re-check watchable phases, reap stray
  dev servers.
- **Run cockpit** - the machine as an ASCII graph (reject arcs, off-path
  branches, live status) or a compact rail; deliverables, the raw activity
  feed, the state history, and the Conductor in tabs. Approve / request
  changes / pause / re-run / route / rewind / re-check, edit the goal, pin
  notes that feed the Conductor.
- **Loops** - browse, duplicate, or AI-draft a loop from a description; edit
  every state (prompts open in `$EDITOR`, tools are a multi-select, gates /
  loop-backs / model pins inline) with a live graph preview; chat-edit with AI.
- **New run** - repo, loop, goal, models, gate mode, run mode, optional-phase
  skips.

`--password` (or `ATELIER_PASSWORD`) unlocks an `ACCESS_PASSWORD`-gated
instance. Your `--user` name signs comments and shows up in presence, same as
the web. `?` shows the keys on any screen; prefs persist in
`~/.config/atelier/cli.json`.

## Ways to run it

Same app in every case - what changes is where the server lives and what
points at it.

- **Solo, on your laptop.** The default: `npm start`, open
  `http://localhost:7777`. No password, nothing exposed.
- **From your phone.** Run the factory on your Mac, put both devices on a
  [Tailscale](https://tailscale.com) tailnet (or any VPN), and open
  `http://<your-mac's-tailnet-name>:7777` from the phone. Flip the top-bar
  switch to **Simple** (saved per device). Set `FACTORY_PUBLIC_HOST` to the
  tailnet name so **Test it** URLs are tappable too.
- **With your team, from your machine.** Password-gate it, then hand out the
  LAN address or a tunnel URL (`cloudflared`, `ngrok`). See below.
- **On a shared, always-on host.** A Mac mini, office box, or VM runs the
  factory as a service; laptops and phones are clients. Provisioning, systemd,
  headless notes: [deploy.md](deploy.md).
- **Terminal-first.** Leave `atelier` full-screen on a monitor - against your
  own factory or anyone else's (`atelier --url http://alice.local:7777`).

## Running it as a team

Atelier is one shared instance, not one install per person. Whoever owns a
loop runs it on the machine that has the repos, the `gh` login, and the model
keys; comments, presence, and the Conductor live server-side, so teammates
install nothing.

1. **Gate it, then share it.** Set `ACCESS_PASSWORD` and
   `FACTORY_INTERNAL_SECRET` in `.env.local` (`setup.sh` generates the
   secret). The server listens on all interfaces; on a shared network the
   board is `http://<your-ip>:7777`, and for remote teammates a tunnel goes in
   front. Visitors unlock once with the password.
2. **Everyone signs their work.** Each person sets a display name (the web
   asks; the CLI takes `--user`). Comments carry their author, presence shows
   who's on a run, and the Conductor reads the feedback by name when it routes.
3. **Review from anywhere; work happens on the host.** Teammates pin comments,
   approve gates, send phases back, talk to the Conductor - all of it executes
   on the host: worktrees on the host's repos, PRs from the host's `gh`
   account, models on the host's keys. A teammate's "approve" runs as the
   owner.

If several people own loops, each runs their own instance and shares it the
same way - boards don't federate. The terminal UI makes hopping between them
cheap.

Worth knowing before you share:

- **One password, one tier.** Anyone with it can do anything - start runs,
  edit loops, reroute, archive. Display names are labels, not accounts. Share
  the password like you'd share push access.
- **The host has to stay awake.** Runs, the queue, and presence live in that
  one process; a sleeping laptop is a paused factory. Runs serialize by
  default (`FACTORY_MAX_CONCURRENT_RUNS=1`).
- **Test-drive opens a door.** The "Test it" dev server binds all interfaces
  and is *not* behind `ACCESS_PASSWORD` - treat it as visible to your network
  while it's up (Reap closes strays).
- `@mentions` highlight a name; they don't notify anyone (yet).

## How it works

- **One agent, whole goal, on a real repo.** Each loop cuts a git worktree off
  a base branch; the agent makes real edits and commits there, and the final
  state opens a PR.
- **The agent harness.** Every model drives the full Claude Code CLI (30+
  tools, real context management). Atelier-specific actions go through a small
  CLI bridge (`bin/factool.mjs`).
- **Non-linear.** Review, Rigorous QA, and Acceptance can **reject** and loop
  work back to Build (bounded by a configurable loop cap), then it comes
  forward through the checks again. The agent gives the verdict; the runner
  only routes.
- **Rigorous QA.** Boots the dev server, logs into auth-walled apps with the
  repo's own test fixtures, decomposes the feature into concrete cases, tests
  each in series, and vision-verifies the captures (did the animation move?
  did the filter change the look?). Each case surfaces as its own
  `Case N - … - PASS/FAIL` deliverable.
- **The Conductor.** A configurable per-loop agent that reads every open
  comment - investigating the actual workspace, not just the words - and
  proposes the earliest phase to re-enter with a revision brief. Two
  independent switches: **Route** (propose waits for your ok / auto routes
  itself) and **React** (reads each comment as it lands / batches until you
  say review). You can talk to it directly, and revert its last route.
  Feedback is decoupled from the SDLC: it can land on a done loop and reopen it.
- **Self-learning.** With `selfLearn` on, feedback distills into
  **principles** - durable, generalized rules stored on the loop and injected
  into every future run (each run can opt out). The agent can record its own
  via `record_principles`. If a text rule can't close a structural gap,
  `evolveStructure` (off by default) lets reflection add a new phase to the
  loop - capped and guarded.
- **Watchable end states.** A terminal phase can declare itself `watchable` -
  Open PR keeps watching CI and rebase status via `watch_pr`, and stays
  re-checkable from the board after the run is done.
- **Deliverables at a glance.** Verdicts colored pass/fail, grouped by
  category, segmented by attempt - fails first. A phase can declare a **case
  input** (feed QA one more scenario mid-run) or a downloadable **output** file.
- **A clean PR.** Open PR rebases onto the latest base and resolves conflicts
  (keeping both intents) before pushing, so the PR merges cleanly.
